# Frontend Fundamentals — 좋은 프론트엔드 코드의 4가지 기준

좋은 프론트엔드 코드는 **변경하기 쉬운** 코드다.
코드를 설계, 구현, 리팩토링할 때 아래 4가지 원칙을 항상 따른다.

---

## 1. 가독성 (Readability)

읽는 사람이 한 번에 고려해야 할 맥락을 최소화하고, 위에서 아래로 자연스럽게 읽히는 코드를 작성한다.

**규칙:**
- 동시에 실행되지 않는 코드는 별도 컴포넌트로 분리한다
- 복잡한 조건식(`a && !b || c`)에는 의미 있는 이름을 붙인다: `const canPurchase = ...`
- 구현 상세는 추상화하고, 핵심 흐름이 드러나게 한다
- 중첩된 삼항 연산자 대신 `if`문이나 early return을 사용한다
- 코드를 위아래로 왔다갔다 읽어야 하면, 조건을 한눈에 볼 수 있는 객체로 관리한다
- 범위 조건은 수학처럼 왼쪽→오른쪽 순서로 작성한다: `80 <= score && score <= 100` (O) / `score >= 80 && score <= 100` (X)
- Hook이나 함수를 로직 종류(쿼리파라미터, 상태, API)로 묶지 않는다. 기능/책임 단위로 분리한다. `usePageState()` 하나에 모든 쿼리 파라미터를 넣지 말고, `useCardIdQueryParam()`처럼 책임별로 나눈다

**하지 말 것:**
- 간단한 로직까지 과도하게 추상화하지 않는다
- 나쁜 코드에 주석을 다는 대신 코드 자체를 개선한다
- "미래 유연성"을 위해 불필요한 깊은 계층을 만들지 않는다

---

## 2. 예측 가능성 (Predictability)

함수와 컴포넌트는 이름이 암시하는 대로만 동작해야 한다. 숨겨진 동작이 없어야 한다.

**규칙:**
- `getX()`, `fetchX()` 같은 함수에 로깅, 상태 변경 등 숨은 부수효과를 넣지 않는다. 부수효과는 호출부에서 명시적으로 처리한다
- 같은 종류의 함수는 반환 타입을 통일한다. Discriminated Union(`{ ok: true } | { ok: false; reason: string }`)을 활용한다
- 함수 이름과 실제 동작이 일치해야 한다. `calculate()`가 저장까지 하면 `calculateAndSave()`로 분리하거나 이름을 변경한다
- 라이브러리/프레임워크 이름과 겹치는 네이밍을 피한다

**하지 말 것:**
- 부수효과를 제거하는 게 아니라, 명시적인 위치로 이동시킨다
- 지나치게 복잡한 타입을 만들지 않는다

---

## 3. 응집도 (Cohesion)

함께 수정되는 코드는 함께 있어야 한다. 한쪽을 수정했을 때 다른 쪽을 빠뜨리는 실수를 구조적으로 방지한다.

**규칙:**
- 함께 수정되는 파일은 같은 디렉토리에 둔다. 종류별(`components/`, `hooks/`, `types/`)이 아닌 도메인별(`domains/User/`)로 배치한다
- 같은 숫자/문자열이 여러 파일에 흩어져 있으면 공유 상수로 추출한다 (매직 넘버 금지)
- import 경로가 `../../..`처럼 길어지면 파일 배치를 재검토한다
- 타입 정의는 사용하는 코드 가까이에 둔다
- 이벤트 핸들러와 그것을 트리거하는 UI는 가까이 둔다. 핸들러 정의와 버튼이 멀리 떨어져 있으면 별도 컴포넌트로 추출하여 로직과 UI를 함께 묶는다

**중복이 나은 경우 (예외):**
- 두 기능이 다르게 발전할 가능성이 높을 때
- 공유 시 원치 않는 의존성이 생길 때
- 공유 코드가 단순할 때 (5줄 미만)

---

## 4. 결합도 (Coupling)

코드 변경의 영향 범위를 최소화한다. A를 수정했을 때 관련 없는 B가 깨지면 안 된다.

**규칙:**
- props가 3단계 이상 drilling되면 조합 패턴(`children`)을 사용한다
- 하나의 Hook이 5개 이상 값을 반환하면 단일 책임 Hook으로 분리한다
- 모든 곳에서 import하는 util은 사용처 가까이로 이동시킨다
- Context는 진정한 전역 관심사(테마, 로케일, 인증)에만 사용한다. props drilling을 피하려고 Context를 남용하지 않는다

**중복이 나은 경우 (예외):**
- 페이지마다 동작이 달라질 여지가 있을 때
- 공통 코드 수정 시 모든 의존 코드 테스트 부담이 클 때

---

## 원칙 간 충돌 시 판단 기준

4가지를 모두 동시에 만족하기는 어렵다. 충돌 시:

- **응집도 vs 가독성**: 함께 수정하지 않으면 버그가 날 수 있는 경우 → 응집도 우선. 위험성이 낮으면 → 가독성 우선 (중복 허용)
- **결합도 vs 응집도**: 공통화하면 영향 범위가 넓어지는 경우 → 결합도 우선 (중복 허용)
- 항상 **현재 상황의 위험도와 변경 빈도**를 기준으로 판단한다
