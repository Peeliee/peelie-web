## 0. 실행 프로토콜

모든 요청에 대해 다음 순서로 행동한다.

1. **작업 분류** — 코드 작성/수정(§2,§3,§4,§5) · 리뷰(§3,§4,§5) · 버그(§3.5 4-step 필수).
2. **트리거 스캔** — 대상 파일을 `grep`으로 §4 표와 매칭. 매칭되면 발동.
3. **자가검열** — 제안 직전 §5 Bad Fix Blacklist 대조. 하나라도 걸리면 다른 방법으로.
4. **완료 선언 전** — 요청된 변경만 포함? 타입 에러 0? `.skip`/`@ts-ignore` 신규 0? PII 위반 없음? 원인 수정이지 증상 회피 아님? 다섯 개 다 통과해야 보고.

### 금지

- 규칙이 불편하다고 **"완화 해석"** 하기. 예외는 §6에 열거된 것만.
- 일반 지식으로 리뷰 출력. 반드시 이 파일 기준.
- `// removed` `// legacy` 같은 추적 주석. 지울 거면 깨끗이.
- 사용자가 요청하지 않은 리팩토링 끼워 넣기.

## 1. 가치 우선순위 (충돌 시)

정확성·타입 안전성 > 응집도 > 예측 가능성 > 결합도 > 가독성 > 간결성.

- 가독성 vs 응집도: 같이 안 고치면 **버그**면 응집도 우선.
- 응집도 vs 결합도: 독립 진화 **여지**가 있으면 중복 허용(결합도 우선).

## 2. 메타 DO / DON'T

| DO                                    | DON'T                                          |
| ------------------------------------- | ---------------------------------------------- |
| 근본 원인 수정                        | 증상 가림 (`as`, `!`, `\|\| {}`, catch 삼키기) |
| `unknown` + 타입가드                  | `any`                                          |
| 절대 경로 (`@/...`)                   | 깊은 `../../..`                                |
| 구조 변경 / 동작 변경 커밋 분리       | 한 커밋에 섞기                                 |
| API 응답 원형 사용                    | 프론트 편의 매핑으로 정보 손실                 |
| 실패 테스트는 고치거나 삭제           | `.skip` / `xit` / 주석 처리                    |
| 중앙 에러 핸들러                      | 곳곳에서 alert                                 |
| 로그에 PII 마스킹                     | 이름·메일·전화·결제 평문                       |
| 주석은 "왜"만                         | "무엇" 주석 (이름으로 표현될 내용)             |
| 기존 파일 수정으로 가능한지 먼저 확인 | 기본값으로 새 파일 추가                        |

**주석 정책 (엄격):** 허용되는 주석 = 복잡한 알고리즘의 수학적 근거 / 라이브러리·브라우저 버그 워크어라운드 / 반직관적 최적화 이유 / 외부 스펙(RFC·W3C) 참조. **그 외 전부 삭제.** `--no-verify`는 사용자 명시 요청 시에만.

## 3. 도메인 규칙

### 3.1 가독성 (Readability)

**기준:** 한 번에 올리는 맥락 ≤ 6~7개, 흐름은 위→아래.

| DO                                                     | DON'T                                                         |
| ------------------------------------------------------ | ------------------------------------------------------------- |
| 동시에 실행 안 되는 분기 → 서브 컴포넌트 분리          | `if (role==='viewer')` 같은 분기 3곳 이상 흩어짐              |
| 복잡 조건에 `const isXxx = ...` 이름 부여              | `filter(x=>x.a.some(y=>... && x.b.some(z=>...)))` 인라인 체인 |
| 숫자·문자는 상수화 (`ANIMATION_DELAY_MS=300`)          | `delay(300)`, `setTimeout(fn, 3000)` 하드코딩                 |
| 범위 조건 수학 부등식 순서: `min<=x && x<=max`         | `x>=min && x<=max` (중간값 반복 확인)                         |
| 중첩 삼항 → `if` / early return                        | 2단계 이상 삼항 `a?b:c?d:e`                                   |
| 구현 상세(`status==='LOGGED_IN'`)는 Wrapper/HOC로 숨김 | 컴포넌트 본문에 도메인 상태 머신 노출                         |

**Triggers:** 2단 이상 인라인 삼항 · 로직 내 3자리 이상 매직 넘버 · JSX 내 조건 분기 3개 이상 · 함수 body > 50줄 + `if/else` 3개 이상 · 시그니처 경계 넘는 `data/info/item/tmp`.

**예외:** 한 번만 쓰이는 단순 식(`arr.map(x=>x*2)`)은 이름 안 붙임. 성능 크리티컬 구간의 의도적 inline은 주석이 있으면 유지.

---

### 3.2 예측 가능성 (Predictability)

**기준:** 함수는 이름 + 시그니처만으로 동작 설명 가능해야 한다.

| DO                                                                | DON'T                                     |
| ----------------------------------------------------------------- | ----------------------------------------- |
| 같은 종류 훅은 반환 타입 통일 (모두 `UseQueryResult`)             | 어떤 훅은 Query 객체, 어떤 훅은 `.data`만 |
| 검증 함수는 `{ok:true} \| {ok:false, reason}` Discriminated Union | `boolean` / `{ok,reason}` 혼재            |
| 부수 효과는 호출부 명시                                           | `fetchBalance()` 내부에서 암묵적 로깅     |
| 이름 겹치면 `httpService.getWithAuth()`처럼 차별화                | 내부 `http.get`이 토큰 주입을 몰래 함     |
| `calculateX`는 계산만, 저장하면 `calculateAndSaveX`               | `get*`/`fetch*`가 캐시를 mutate           |

**Triggers:** 함수명 `get/fetch/calculate/parse/format/isXxx`인데 body에 `setState`/`localStorage`/`logger`/`analytics`/`track` · 같은 폴더에 `Foo`/`Foo\|null`/`{data:Foo}` 반환이 혼재 · 로컬 심볼이 서드파티 동명 import를 섀도잉.

**예외:** 관측 가능성이 필수인 경우 함수 이름에 `Tracked` / `WithAnalytics` 접미사로 명시.

---

### 3.3 응집도 (Cohesion)

**기준:** 함께 바뀌는 것은 같은 디렉토리·파일·상수에 있다.

| DO                                                   | DON'T                                                |
| ---------------------------------------------------- | ---------------------------------------------------- |
| 도메인 배치: `domains/Cart/{components,hooks,utils}` | 타입 배치: 전역 공용 `components/`·`hooks/`·`utils/` |
| 같은 의미 상수는 한 출처                             | 파일마다 `const DELAY=300` 복붙                      |
| 필드 검증 독립 → 필드 단위 응집                      | 전부 Form에 몰아두고 재사용한다 주장                 |
| 필드간 의존·순서 있음 → 폼 단위 응집                 | 필드 Validator들이 서로 import로 얽힘                |

**Triggers:** 한 기능 수정에 3종류 타입 폴더 동시 편집 · `git log --follow`상 두 파일이 90% 이상 동시 커밋 · `../../..` 경로 · 3파일 이상 동일 값 중복 상수.

**예외 (중복 허용):** 독립 진화 가능성 / 공유 코드 < 5줄 + 의존성 역전 유발 / 페이지별 변주 여지(로그 키·A/B 텍스트).

---

### 3.4 결합도 (Coupling)

**기준:** A 고쳤을 때 무관해 보이는 B가 깨지지 않는다.

| DO                                                         | DON'T                                        |
| ---------------------------------------------------------- | -------------------------------------------- |
| Props Drilling 3계층 → **Composition(`children`) 먼저**    | 곧바로 Context/Redux/Zustand 도입            |
| 단일 책임 Hook (`useCardIdParam`, `useDateRangeParam`)     | `usePageState` 하나가 파라미터·API·로그 전부 |
| Context는 진짜 전역만 (테마·로케일·Auth·10+ 컴포넌트 공용) | 부모-자식 2단계 회피용 Context               |
| 공통화는 최소 3회 반복 확인 후                             | 두 번째 유사 코드에서 즉시 `useShared...`    |

**Triggers:** 훅이 5개 이상 값 반환 · 같은 이름 prop이 3계층 이상 통과 · 테스트에서 mock 의존성 5개 이상 · 한 수정이 `tsc` 에러를 3파일 이상으로 터뜨림.

**예외:** 실험·마이그레이션 스크립트는 신경 안 쓴다. 리렌더 최적화 목적의 단일 책임 쪼개기는 과하지 않음.

---

### 3.5 디버깅 (Debugging) — 버그 요청 시 필수 프로토콜

**순서 건너뛰기 금지:** 진단 → 재현 → 수정 → 재발 방지.

1. **진단** — 에러 분류(Syntax/Type/Reference/Resource/Module). 작업 지도(입력→상태→이펙트→출력). 최초 실패 지점과 최초 관찰 시점 분리.
2. **재현** — 재현 못 하면 아직 못 고친다. 최소 케이스까지 축소. 경계값 ±1 / 빈 배열 / 음수 / `MAX_SAFE_INTEGER`. 도구: `console.table/group/time/trace/assert`, tagged `[Module]`.
3. **수정 — 원인만 고친다:**

| 증상 가림 (금지)                          | 근본 수정                                    |
| ----------------------------------------- | -------------------------------------------- |
| `user as User`, `user!.name`              | `if (!user) throw ...`                       |
| `value ?? {}` / `\|\| []` 무조건 폴백     | 무효 상태를 **타입으로 배제**                |
| `setTimeout`+`clearTimeout`으로 race 가림 | `AbortController` + 디바운스                 |
| `try{}catch(e){}`                         | 복구 가능한 것만 잡고 나머지 전파            |
| `useEffect(..., [])` + eslint-disable     | 의존성 정직하게; 필요 시 ref/함수형 업데이트 |
| 실패 테스트 `.skip`                       | 테스트가 잘못이면 고치고, 맞으면 코드 고친다 |

4. **재발 방지** — 타입 / ESLint 룰 / 테스트 / 어설션 중 **최소 1개** 추가. 데드 코드(과거 A/B·임시 플래그) 정리. 로그에 **PII 절대 금지**.

**PR 본문 필수 4항:** 재현 환경 · 재현 흐름 · 기술적 원인 · 근본 해결 · 재발 방지 수단.

## 4. Refactoring Triggers (grep 빠른 스캔)

파일을 건드릴 때 한 번 훑는다. 매칭되면 같은 PR에서 함께 정리(허용되면) 또는 후속 이슈.

| 트리거                                                 | 규칙      | 행동                    |
| ------------------------------------------------------ | --------- | ----------------------- |
| `\? .+ \? ` 인라인 중첩 삼항                           | §3.1      | `if`/early return       |
| 로직 내 3자리 이상 매직 넘버                           | §3.1/§3.3 | 상수화                  |
| `get/fetch/calculate*` body에 `analytics/logger/track` | §3.2      | 부수 효과 분리          |
| 훅이 5+ 반환 또는 이름에 `PageState`/`AppState`        | §3.4      | 단일 책임 분해          |
| 같은 prop 3계층 이상 통과                              | §3.4      | Composition             |
| 상대 경로 `../../..`                                   | §3.3      | 가까이 이동 / alias     |
| 전역 `components/`·`hooks/`·`utils/` 폴더              | §3.3      | 도메인 기반 재조직 제안 |
| 서드파티와 동명의 로컬 심볼                            | §3.2      | 서비스 이름으로 변경    |
| `a >= b && a <= c`                                     | §3.1      | `b <= a && a <= c`      |
| `.skip`/`xit`/`@ts-ignore`/`eslint-disable`            | §2/§5     | 원인 고치거나 삭제      |

## 5. Bad Fix Blacklist — LLM이 흔히 저지르는 나쁜 수정

**제안 직전 이 리스트를 통과시킨다. 하나라도 걸리면 폐기하고 다시 생각.**

1. 과도 추상화 — 문제 해결용 팩토리/빌더/Strategy 신설("미래 확장성" 핑계 포함).
2. 주석 회피 — 코드는 그대로 두고 `// 이건 이렇게 동작함` 만 추가.
3. 타입 단언 남발 — `as unknown as Foo`, `!`로 에러 끄기.
4. 에러 삼키기 — `catch(e){}` 또는 `console.error(e)`만 찍고 계속 진행.
5. 의존성 배열 거짓말 — `useEffect(..., [])` + `eslint-disable`.
6. 전역 상태 남용 — 2단계 prop 회피용 Context/Redux/Zustand 도입.
7. JSDoc만 추가 — 동작은 그대로, 문서만 덧붙임.
8. 가드만 추가 — race에 `isMounted` 플래그, `AbortController` 미도입.
9. 타입 가드만 추가 — 일관 반환으로 통일 안 하고 호출부마다 좁히기.
10. 단일 거대 파일 — 응집도 핑계로 1000줄 파일 신설.
11. 중복 상수 병렬 생성 — 파일마다 같은 상수, 각자의 진실.
12. God hook에 옵션 플래그 추가 — `options.mode='viewer'|'admin'` (정답: 분리).
13. PII 로깅 — 이메일·전화·결제 평문 Sentry/로그.
14. 원인 파악 없이 덮어쓰기 — "일단 되게 함" 커밋.
15. 요청되지 않은 리팩토링 끼워 넣기 — 버그 PR에 포맷·이름·폴더 섞임.

## 6. 예외 / Escape Hatches

규칙을 어기려면 **코드 옆에 이유 한 줄**. 없으면 불인정.

| 허용                | 조건                                                          |
| ------------------- | ------------------------------------------------------------- |
| 삼항 중첩           | 2단계 이하 + 한 줄 + JSX 내부 + 의미 명확                     |
| 매직 넘버 유지      | 수학 공식의 `0/1/2` 같은 자명한 수                            |
| 중복 유지           | §3.3 예외 충족 (독립 진화 / <5줄+의존성 역전 / 페이지별 변주) |
| 타입 폴더 배치      | 프로젝트 초기 + 기능 ≤ 3개                                    |
| `any`               | 서드파티 타입 미비 + `@ts-expect-error` + 이슈 링크           |
| `--no-verify`       | 사용자 명시 요청                                              |
| `as` assertion      | 런타임 스키마 검증이 타입 시스템보다 먼저 수행되는 경우       |
| Context             | §3.4 "진짜 전역" 기준 충족                                    |
| 함수 내부 로깅 유지 | 함수 이름 `...Tracked`/`...WithAnalytics` 명시                |

## 7. 자가 테스트 (로드 직후)

두 개를 즉답 못 하면 이 문서를 다시 읽어라.

1. `useEffect(()=>{fetch().then(setX)},[])` 빠른 입력에 stale data → §3.5 race. `AbortController` 사용.
2. `useCartPageState()` 가 `cartId/coupon/address/shipping/memo/isLoading/error` 반환 → §3.4 god hook. 책임별 분해.

**끝.** 규칙을 우회할 이유를 찾지 말고 따를 방법을 찾는다. 규칙 자체가 틀렸다면 조용히 어기지 말고 이 파일을 PR로 고친다.
