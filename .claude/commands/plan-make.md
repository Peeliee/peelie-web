# 계획서 생성 워크플로우

사용자의 요청을 바탕으로 작업 계획서를 생성하고 발전시킨다.

## 실행 절차

### 1. 계획서 파일 생성

1. `plan/YYYY-MM-DD/` 디렉토리를 찾는다. 없으면 생성한다. (오늘 날짜 기준)
2. 해당 디렉토리에 이미 있는 파일 수를 세어 다음 번호를 결정한다 (00, 01, 02, ...). `-deprecated` 파일도 카운트에 포함한다.
3. 파일명: `{번호}-{한국어-요약-제목}.md` (예: `00-인증-시스템-리팩토링.md`)
4. 제목은 작업 내용을 한국어로 간결하게 요약한다.

### 2. 계획서 구조

```markdown
# {제목}

## AS IS (현재 상태)
- 현재 코드/시스템 상태를 명확하게 기술
- 근거가 되는 코드, 파일 경로 등을 구체적으로 포함

## TO BE (목표 상태)
- 달성하고자 하는 목표 상태를 명확하게 기술
- 측정 가능한 기준이 있으면 포함

## 구현 내용
- 설계, 구현 방향, 기술적 결정 사항 등
- 필요 시 OMC planner 에이전트(`oh-my-claudecode:planner`)를 활용하여 설계를 구체화한다
```

**AS IS**와 **TO BE**는 반드시 아주 명확하게 작성한다.

### 3. 계획 디벨롭 단계 (작업 시작 전)

- 사용자가 **"작업 시작"**이라고 하기 전까지는 **계획서 파일만 수정**한다.
- 소스 코드 파일은 절대 수정하지 않는다.
- 문제 정의를 더 명확히 하고, 구현 방향과 설계를 구체화하는 데 집중한다.
- 사용자와 대화하며 계획서를 반복적으로 개선한다.
- 복잡한 설계가 필요하면 OMC planner(`oh-my-claudecode:planner`)나 architect(`oh-my-claudecode:architect`) 에이전트를 활용한다.

### 4. 작업 완료 처리

- 사용자가 **"작업 완료"**라고 하면, 해당 계획서 파일명 끝에 `-deprecated`를 추가한다.
- 예: `00-인증-시스템-리팩토링.md` → `00-인증-시스템-리팩토링-deprecated.md`
- 이를 통해 완료된 계획과 미완료 계획을 구분한다.
